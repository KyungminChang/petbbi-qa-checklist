# baseline(product_spec_petbbi_v1.md)이 아니면 output/qa-checklist-<식별자>.csv 로 자동 분리 생성
# 단계별로 점검하고 싶을 때만 /analyze → /insight → /generate → /review → /validate 순서로 개별 실행
체크리스트 양식 설계 근거 (본인 설계) QA 담당자는 "어떤 케이스를 먼저 실행할지"를, PM은 "출시를 막을 P0가 무엇인지"를 본다. 그래서 컬럼을 테스트유형·우선순위·출처 3축으로 고정해 두 관점을 한 표에서 필터링할 수 있게 했다. 출처(기획서기반/AI추가) 컬럼은 "기획서 자체의 빈틈"을 가시화해 다음 기획서 작성 품질로 피드백된다.
테스트유형 분류 기준 기획서에 명시된 정상 흐름 → Happy Path / 기획서에 명시된 오류 처리 → Unhappy Path / 기획서에 아예 없는 예외(빈 상태·권한·네트워크·경계값·AI 폴백·명세 미정의) → Edge Case(출처는 원칙적으로 AI추가). 다만 TC-004·TC-017처럼 명세에 문장으로 존재하는 경계값은 Edge Case이면서도 출처=기획서기반인 예외가 있어, 테스트유형만으로 출처를 100% 예측할 수는 없다.
/validate가 하는 일 (Standard에서 추가한 5단계) /review가 "품질 기준을 충족했는가"를 본다면 /validate는 "데이터 자체가 정확한가"를 기계적으로 재검산한다 — CSV 컬럼 수·중복·ID 오름차순, 그리고 리포트·decisions.md에 적어둔 수치(82건·P0 15건 등)가 실제 CSV와 일치하는지 대조한다.
